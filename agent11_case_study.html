The Problem

Most recommendation systems are opaque — they surface products via collaborative filtering or embedding similarity without telling the customer why. The result is a feed that feels generic, loses trust when it misses, and gives product teams no signal to improve on.

The harder problem is cold-start: new or returning customers without rich behavioural data still deserve relevant recommendations. Waiting for enough interaction data to feed a model is not a product strategy.

Core insight: A brief, well-structured conversation is a faster and more reliable signal than weeks of passive behavioural observation — and it produces explainable, auditable reasoning rather than a latent embedding.

How It Works

The agent uses a structured conversation flow to elicit the signals that matter — use case, constraints, preferences, and budget — before generating recommendations. Each recommendation is accompanied by conditional reasoning that maps directly back to what the customer said.

01
Domain discovery
Open-ended first turn establishes category and intent without constraining the customer.
02
Preference elicitation
1–2 focused follow-ups on budget, use case, and hard constraints. Conversation triggers recommendations as soon as sufficient signal exists.
03
Conditional reasoning
Each recommendation is generated with explicit if/then logic: "Given X constraint and Y preference, this fits because Z."
04
Ranked output
2–3 products surfaced in priority order with structured justification. Customer can ask follow-ups to refine.

Design Decisions

Explainability First

Justifications are generated inline, not post-hoc. The reasoning is part of the output format, not an afterthought.

Domain Agnostic

No hard-coded product categories. The agent discovers domain through conversation and adapts its criteria accordingly.

Minimum Turns

System prompt enforces recommendation trigger once sufficient context exists — no unnecessary back-and-forth.

Structured Output

Delimited product blocks parsed client-side into clean cards — LLM output and UI rendering are fully decoupled.

System Prompt Architecture

The agent's behaviour is governed by a compact system prompt (under 300 tokens) that encodes conversation flow, output format, and reasoning style. A delimiter pattern lets the client parse structured product data out of a free-form completion without a separate tool call or JSON mode — keeping the single API call constraint intact. 

// Output format — delimiter-based structured extraction
===PRODUCT===
NAME:  [product name]
TAG:   [category]
DESC:  [2–3 sentence justification tied to stated preferences]
===END===

// Conversation state machine
turn 1    → open discovery question
turn 2–3  → budget / use-case / constraints
turn 3–5  → trigger recommendations if context sufficient
post-rec  → handle follow-ups, surface alternatives

Extension Path

The current implementation uses illustrative product generation — appropriate for demonstrating the reasoning layer. The production extension is direct: replace illustrative generation with a real-time product catalogue lookup via web search or internal vector store, with Claude acting as a reasoning and ranking layer over live inventory.

The conversation history structure also maps cleanly onto a persistent profile store — each session can contribute incrementally to a customer embedding that informs future cold-start behaviour without requiring explicit re-profiling.

Retail applicability: The explainability layer is directly useful for merchandising teams who want to understand why products are being surfaced — enabling feedback loops that black-box models can't provide.
