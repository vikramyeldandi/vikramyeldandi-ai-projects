Agent 16 · Multi-Agent Marketing Repurposer

Six Agents, One Run, One Campaign Pack

How an orchestrator-first multi-agent architecture with parallel specialist execution and a synthesiser quality pass turns a single piece of content into a consistent four-channel campaign — without a marketing team.

The Problem

Marketing teams write one good piece of content and then manually rewrite it four or five times for different channels. The blog post becomes an email, the email becomes a LinkedIn post, the LinkedIn post gets compressed into tweets, and somewhere along the way the core message drifts. The Product Hunt tagline contradicts the email subject line. The Twitter thread uses a statistic the LinkedIn post doesn't mention. By the time all four channels are published, the campaign lacks the consistency that makes it memorable.

The problem is not creativity — it's coordination. Each format has different constraints, different audiences, and different conventions. Doing it well requires either a specialist for each channel or a generalist who context-switches four times and loses fidelity each time.

Architecture

Agent 16 uses a three-stage pipeline: orchestrator, parallel specialists, synthesiser. The key architectural decision is that specialists never see the raw source content — they only receive the orchestrator's structured brief. This is not a performance optimisation. It's a quality decision.

Why specialists receive a brief, not the source: If four agents all read a 600-word blog post independently, they each extract different insights, emphasise different angles, and produce outputs that diverge. When they all work from the same structured brief — same core message, same key points, same CTA — divergence becomes a choice rather than an accident. The synthesiser then has a meaningful job: checking whether the choice was sound.

Orchestrator — Call 1

Reads the source content and produces a structured brief: core message, audience, tone, key points, CTA, and a list of things to avoid. This brief is the only input specialists receive. The orchestrator runs alone — its output quality gates everything downstream.

Specialists — Calls 2–5 (parallel)

Four agents fire simultaneously via Promise.all: Email, LinkedIn, Twitter/X, and Product Hunt. Each has a distinct system prompt with its own persona, constraints, and output schema. They receive identical input but produce structurally different outputs.

Synthesiser — Call 6

Receives all four outputs plus the original brief and runs a consistency check. Outputs a Pass or Flag verdict with specific checks on message alignment and tone. If it flags an inconsistency, the UI surfaces it explicitly rather than hiding it.

Pipeline Visualiser

The UI renders a live agent pipeline showing each node activating and completing in real time. The six nodes — ORC, EM, LI, TW, PH, SYN — pulse as their call fires and turn solid when done. This makes the multi-agent architecture visible, not just implied.

Channel Constraints

Each specialist operates under explicit format constraints enforced by its system prompt. These constraints are not arbitrary — they reflect the real publishing requirements of each platform and force the model to make editorial decisions rather than padding to fill space.

Email

Subject line capped at 55 characters. Preview text at 90 characters. Body at 180 words, prose only — no bullet points. Opens with the insight, not a greeting. One CTA button, max 4 words.

LinkedIn

150–200 words. Hook first — not a question. Max 2 sentences per paragraph for readability. One concrete data point. Max 3 hashtags. Written as a VP would post, not a social media manager.

Twitter / X

4-tweet thread, each under 240 characters. Every tweet must earn its place — no filler. One hashtag maximum, on the final tweet only. The hook tweet is the most constrained: it must stop a scroll.

Product Hunt

Tagline at 60 characters — what it does, not what it is. Description at 260 characters. Maker comment written in first person as the founder, 80–120 words. Genuine, not polished.

What I Learned

The orchestrator is the most load-bearing prompt in the system. If the brief is vague, all four specialists produce vague output. The orchestrator prompt went through the most iteration of any in this project — specifically the AVOID field, which prevents specialists from inventing angles or statistics the source content doesn't support. Without it, specialists hallucinate specifics to fill their formats.

Parallel execution changes the UX contract. When four calls fire simultaneously, the user waits for the slowest one — typically 3–5 seconds. This is faster than sequential execution but long enough to need clear status signalling. The pipeline visualiser with pulsing nodes is not decoration — it's the feedback mechanism that tells the user something real is happening and prevents abandoned sessions.

The synthesiser is more useful as a flag than a fixer. An earlier design had the synthesiser rewrite inconsistent outputs. This made the pipeline 8 calls instead of 6 and added latency without proportional quality improvement. The current design surfaces flags explicitly and lets the user decide what to do — which is the right allocation of human judgment in an agentic workflow.

Brand context must be in every call, not just the orchestrator. In early testing, specialists drifted toward generic marketing language when they only received the brief. Adding the Arcflow brand context — tone, audience, what to avoid — directly to each specialist's system prompt anchored the voice consistently across all four channels.

Typical Token Profile

A 400-word source article processed across six agents typically produces:

Orchestrator: ~600 tokens Each specialist: ~700 tokens Synthesiser: ~900 tokens Total: ~4,100–5,300 tokens / run
Cost per run: At Sonnet pricing ($3/MTok input, $15/MTok output), a full 6-agent run costs approximately $0.015–0.022 — less than two cents. The parallel execution of four specialists means the wall-clock time is roughly equivalent to two sequential calls, not five.
