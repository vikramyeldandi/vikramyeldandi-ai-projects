01 — Problem & Objective

Why manual catalogue work is a bottleneck

Retail teams spend significant time manually writing product descriptions and tagging attributes across large catalogues. For a brand like Meridian Athletic with hundreds of SKUs across apparel, footwear, and gear, this is slow, inconsistent, and error-prone — especially when images are the primary source of truth for new arrivals.

The objective was to build a multimodal agent that accepts product images and returns structured, retailer-ready output — not a generic image caption, but a validated JSON object with named attributes, quality flags, and an overall quality verdict that a merchandiser or catalogue manager can act on immediately.

The Problem

Product catalogues require consistent descriptions, accurate attribute tagging, and quality control at scale. Manual processes create bottlenecks and inconsistency — especially in fast-moving retail environments where new inventory arrives constantly.

The Objective

A single-call vision agent that returns a strict JSON schema — product name, category, description, five named attributes, typed quality flags, and an overall quality verdict — from a product image with optional category and brand context.

02 — Structured Output Schema

What the agent returns for every image

Rather than free-form text, the agent is constrained to return a defined JSON schema — enforced through the system prompt. Each field serves a specific downstream use case: descriptions feed PDP copy, attributes populate filter systems, quality flags route images to review queues.

📋 product_name
Inferred product name

A concise, retailer-appropriate product name derived from visual analysis — type, style, and key distinguishing feature.

🏷️ category
Classified product category

The model's visual classification of product type — confirmed or corrected against the optional category hint provided at input.

✏️ description
Retailer-ready 2–3 sentence copy

A product description written for a PDP or catalogue — factual, benefit-led, and ready to use with minimal editing.

📐 attributes
Five named visual attributes

Color, material, style, condition, and target audience — extracted from the image and structured for faceted search and filter systems.

⚠️ quality_flags
Typed positive / caution / negative flags

Observations about image or product quality — well-lit background, visible defect, poor contrast, strong lifestyle context. Each flag is typed for colour-coded UI rendering.

✅ overall_quality
Pass / Review / Fail verdict

A three-state verdict for routing — pass goes live, review routes to a merchandiser, fail triggers a reshoot request. Automates QA triage at catalogue scale.

03 — Key Design Decisions

How the agent was architected

Several deliberate constraints shape how Agent 06 behaves — each one a product decision with a clear rationale, not a default technical choice.

1
Single call per image — no pipelines

Each image is analysed in one API call. No multi-step chains, no pre-processing calls. This keeps latency low, cost predictable, and the architecture auditable. The tradeoff is that the system prompt must do more work to constrain output quality.

2
JSON-only output enforced in the system prompt

The system prompt instructs the model to return only a raw JSON object — no markdown fences, no preamble, no explanation. The front end strips any accidental fences before parsing and surfaces a legible error if the model goes off-schema.

3
Context hints injected into the user turn, not the system prompt

Category and brand context are appended to the user message at runtime — not hardcoded into the system prompt. This keeps the system prompt under 300 tokens and makes the agent reusable across brands and categories without re-prompting.

4
Base64 encoding client-side

Images are encoded to base64 in the browser before being sent to the Cloudflare Worker. No server-side file storage, no upload pipeline, no signed URL management. The Worker passes the image content block directly to the Anthropic API.

5
Typed quality flags for downstream routing

Each quality flag carries a type: positive, caution, or negative. This isn't cosmetic — it's designed so that a downstream system can parse the JSON and route images to the correct workflow without reading the flag text. The UI colour-coding is a visible demonstration of that logic.

04 — System Prompt Design

How structured output is enforced

The system prompt is under 300 tokens — deliberately compact. It defines the output schema inline, prohibits any non-JSON response, and leaves no ambiguity about field types. The model has no discretion over format; only content is variable.

System Prompt Structure 
// System prompt enforces schema — user turn carries context

SYSTEM:
You are a product image analyst. Given a product image,
return ONLY valid JSON (no markdown, no prose) with this schema:

{
  "product_name": string,
  "category": string,
  "description": string,  // 2-3 sentences, retailer-ready
  "attributes": {
    "color": string,
    "material": string,
    "style": string,
    "condition": string,
    "target_audience": string
  },
  "quality_flags": [
    { "flag": string, "type": "positive"|"caution"|"negative" }
  ],
  "overall_quality": "pass"|"review"|"fail"
}

// USER TURN appends runtime context:
// "Analyse this product. Category hint: apparel. Context: Meridian Athletic, premium tier."
// Context in user turn → system prompt stays reusable across brands

06 — Lessons Learned

What I would do differently

1

JSON enforcement requires redundancy in the system prompt. Instructing the model to "return only JSON" once is not enough — models occasionally add a preamble or wrap output in markdown fences, especially when uncertain. The prompt needs to state the constraint multiple ways, and the front end needs a fence-stripping fallback before attempting to parse. In production, a JSON mode API parameter (where available) would remove this fragility entirely.

2

Image quality directly affects output quality. The model performs significantly better on clean, well-lit product images against neutral backgrounds. Lifestyle shots with complex backgrounds reduce attribute accuracy — particularly for material and condition fields. In a production catalogue pipeline, image pre-screening (resolution, background complexity) before sending to the vision model would improve overall accuracy.

3

The overall_quality verdict needs calibration, not just instruction. Telling the model to return "pass", "review", or "fail" produces consistent formatting but inconsistent thresholds. Without explicit definitions of what constitutes each verdict — image resolution, background cleanliness, product visibility — the model applies its own judgement. In production, these thresholds should be defined explicitly in the system prompt with examples.

4

Sequential processing is a bottleneck for large batches. Agent 06 processes images one at a time to stay within the single-call-per-interaction constraint. For a catalogue of 50+ images, parallel processing with promise batching would be necessary. The current architecture is correct for a demo and for small batches; production scale requires a queue-based approach with rate limit management.

5

Context hints have a measurable impact on output relevance. Providing a category hint ("apparel") and brand context ("Meridian Athletic, premium tier") in the user turn noticeably improves description tone and attribute precision. Without context, descriptions are generic. This confirms that even single-call vision agents benefit from runtime context injection — a pattern worth standardising across the portfolio.
